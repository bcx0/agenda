## 🔴 Critique

### 1. Double-booking possible (race condition)
`prisma/schema.prisma` (model Booking) + `lib/booking.ts:531-559`. Aucune contrainte d'unicité sur le créneau : la transaction fait SELECT puis INSERT en READ COMMITTED. Deux requêtes simultanées sur le même créneau passent toutes les deux.
**Fix** : index unique partiel côté Supabase :
```sql
CREATE UNIQUE INDEX booking_confirmed_slot_unique
ON "Booking" ("startAt") WHERE status = 'CONFIRMED';
```

### 2. Aucune RLS
`migration.sql` : 0 policy. L'app passe par Prisma/service role côté serveur, mais `NEXT_PUBLIC_SUPABASE_ANON_KEY` est référencée (`lib/supabase/server.ts:6`). Si l'anon key est active, **toutes les tables sont lisibles/modifiables via l'API REST Supabase**.
**Fix** : `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;` sur toutes les tables (sans policy = tout bloqué pour anon, le service role bypass).

### 3. `/api/email/test` public
`app/api/email/test/route.ts` : GET sans authentification → n'importe qui peut vider le quota Resend. Ton email perso est en dur ligne 4. **Fix** : vérif session admin ou suppression de la route.

### 4. Callback OAuth sans `state`
`app/api/auth/google/callback/route.ts` : pas de validation du paramètre `state` (CSRF OAuth) + URLs prod en dur (`agenda-geoffreymahieu.vercel.app` lignes 8, 27, 81) → utiliser `APP_URL`.

### 5. Route export = toutes les données clients
`app/api/admin/export/route.ts` : simple Bearer token statique (`EXPORT_API_TOKEN`), exporte noms/emails/bookings. Token fuité = fuite totale. À minima logger les accès ; idéalement passer par la session admin.

## 📱 Logique PC vs téléphone

Détection : 100% CSS (`md:hidden` / `hidden md:block`). Conséquences :

### Élevé
- **Les deux vues sont montées en même temps** avec des **états indépendants** : `MobileBookingView` a son propre `selectedKey` (`:62`), le desktop son `selectedDay` (`BookingViews.tsx:54`). Resize autour de 768px = sélection perdue/divergente, double exécution des hooks.
- **Logique métier dupliquée** entre mobile et desktop :
  - `getSlotMonthKey()` copiée **4×** (`MobileBookingView.tsx:17`, `BookingViews.tsx:29`, `DaySlotsPanel.tsx:10`, `WeekCalendar.tsx:18`)
  - groupement des slots par jour réimplémenté des deux côtés (`MobileBookingView.tsx:32-47` vs `BookingViews.tsx:199-210`)
  - C'est exactement ce qui a causé tes derniers commits de fix "mobile only" (`0d63ca2`, `7024bb3`) : un fix appliqué d'un côté ne se propage pas à l'autre. **→ extraire dans `lib/timeSlots.ts` et importer des deux côtés.**
- **Groupement par jour en Miami TZ mais clé de quota en Brussels TZ** : un slot qui chevauche minuit peut être compté dans un mois et affiché dans un autre.
- **Comportement différent** : mobile auto-sélectionne le jour courant, desktop n'en sélectionne aucun ; pas de vue "Liste" sur mobile.

### Moyen
- **Admin pas vraiment mobile-first** : `AdminGeneralAvailability.tsx` a deux rendus totalement divergents (`:316` mobile timeline vs `:555` desktop grid) → risque que les données diffèrent ; formulaires `md:grid-cols-*` sans travail mobile réel (`admin/availability/page.tsx:364,480,525,638`).
- **Labels incohérents** : légende mobile "Disponible/Belgique/Complet" vs desktop "Miami/Belgique/Unavailable" ; indicateur Belgique = dot sur mobile, bordure 4px sur desktop.
- `DaySlotsPanel` : `max-w-xl` déborde sous 375px, pas de swipe-down pour fermer.
